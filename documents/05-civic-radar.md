# 05 — Civic Radar

The Civic Radar is a section integrated into each geographic forum with two complementary functions: **inform citizens** about government activity that affects them, and **provide a shared information base** that elevates deliberation quality in the forums. Deliberation without information produces opinions based on assumptions; deliberation with information produces consensus based on facts.

Accessed from one of the three internal navigation blocks of each forum (alongside "Threads" and "Reports"). Two tabs inside the Radar: **"Legislative news"** (default) and **"Government transparency."**

## Component 1 — Legislative and Regulatory News

A feed of laws, decrees, ordinances, official bulletins, and regulations that affect the forum's jurisdiction. Every item is processed by an LLM that generates a **plain-language summary** designed so any citizen can understand what the rule is about without reading the full legal text. See prompt in [14-llm-prompts.md §9.6](14-llm-prompts.md).

### Data sources per pilot city

**CABA (Argentina):**
- Boletín Oficial de CABA: `boletinoficial.buenosaires.gob.ar`
- Legislatura de CABA: `legislatura.gov.ar`
- Open government data: `data.buenosaires.gob.ar`

**San Francisco (USA):**
- DataSF: `data.sfgov.org` (Socrata API)
- SF Board of Supervisors: minutes and ordinances
- California legislation: LegiScan API
- Federal legislation: `congress.gov` API

**Ingestion method** — automated scraping, manual curation, or hybrid — is [19-open-questions.md Q4](19-open-questions.md). At the product level the commitment is: **sources must always be official and verifiable.**

### List view

Items rendered as cards in a vertical list with infinite scroll. Each card shows:
- Rule type (colored badge)
- Title
- Date
- AI summary preview (first 2 lines truncated)
- "Open debate" badge if a thread is already linked

**Filters:** rule type (Law, Decree, Ordinance, Resolution, Official Bulletin, Other), publication date (range with datepicker), debate state (All / With open debate / Without debate), affected jurisdiction.

**Text search** by keyword.

### Item detail screen

On clicking an item:
- Full title and metadata (type, date, jurisdiction).
- **AI summary block** with persistent banner: "Summary generated by AI — Check the original text."
- **"Report error in summary"** button → inline text field (max 500 chars). If a summary accumulates N error reports, it triggers human review.
- Original text block (embedded scrollable text or PDF viewer with download link).
- Linked-debate block.

### Linked-debate mechanic

Each item includes an **"Open debate"** button. When the first user clicks it:

1. A thread is automatically created in the forum at the matching geographic level.
2. The thread is pre-linked to the Radar item.
3. Title is pre-filled and editable (e.g., `Debate: Municipal Ordinance 3421 on education budget`).
4. The Radar item card is embedded at the top of the thread as permanent context.

Once the thread exists, the button changes to **"Go to debate"** for other users, preventing duplication. The thread behaves like any other forum thread and can escalate through the deliberation pipeline if elevation criteria are met.

### Empty state

If no items: illustration + text "No legislative news recorded for this forum yet."

## Component 2 — Government Transparency

Accessed from the "Government transparency" tab.

A dashboard of public-management data showing real information on what the jurisdiction's government is doing. **Goal:** give the citizen direct access to management data that normally goes unnoticed.

### Editorial principle

**The platform presents pure data without qualification.** No judgment of whether a budget allocation is "good" or "bad". The data is shown as it comes from the official source. Interpretation belongs to citizens in the forums, not to the platform. **DemocratIA informs, it does not editorialize.**

### Dashboard content (when data is available)

- **Budget execution** (bar chart by area/department: allocated vs. executed).
- **Public works in progress** (list with name, status, deadline, amount).
- **Tenders and contracts** (list with process, status, awarded amount).
- **Appointments and structure** (org chart or officials-by-area list).
- **Public-service indicators** (when available).

Each visualization has a cited source with link to the official portal. Visualizations are **simple** — basic bar and pie charts (Recharts), **not complex interactive dashboards.**

### Asymmetric availability

Open-data availability varies enormously between jurisdictions. When there is no data, a transparent message appears: **"Management data for [jurisdiction] is not currently available"** with explanation and links to existing official portals (if any).

## Accessibility UX considerations

The Civic Radar is designed to be usable by the entire population, including older adults and people with low digital literacy:

- **AI summaries must be written in simple, everyday language**, not legal or technical jargon. "This law changes how neighborhood complaints to the city are processed" is correct. "This law amends Article 23 of the Administrative Procedures Code" is not useful as a summary (though it may appear as secondary technical reference).
- The Radar is one of the three primary navigation blocks inside each forum, with prominent presence.
- Readable typography, high contrast, minimum number of clicks to reach information or the linked debate.

## Ingestion pipeline (technical)

Run by Vercel Cron every 6 hours:

1. **Fetch** new items from configured sources per jurisdiction (scraping or API).
2. **Deduplicate** by `source_url` or title hash.
3. **Summarize** via Claude API with the Radar summary prompt.
4. **Save** in the `radar_items` table, linked to the forum of the matching jurisdiction.
5. **Notify** users who have Radar notifications enabled for that forum.

See [13-api-and-backend.md](13-api-and-backend.md) for cron-job configuration and [12-database.md](12-database.md) for the `radar_items` schema.
